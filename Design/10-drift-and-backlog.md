# Design — Drift Reconciliation & Pending Backlog

Iterations 1–3 designed an **Event-Sourced** system (iteration 3 closed on "Eventuate ES + Kafka"). From **DD-14** onward the build pivoted to **Eventuate Tram** (state-stored aggregate + transactional outbox). This document reconciles the original iteration 1–3 design against what is actually built, separating **intentional drift** (design changed by a later DD) from **pending backlog** (designed, never built, not superseded). Verified against code on 2026-06-23.

See also: [08-design-decisions.md](08-design-decisions.md) · [06-ott-auth.md](06-ott-auth.md) · [01-system-context.md](01-system-context.md) · [09-product-types-redesign.md](09-product-types-redesign.md).

## 1. Drift — designed in iter 1–3, then changed by a later DD (intentional)

| # | Iter 1–3 design | Current build | Superseded by |
|---|---|---|---|
| 1 | **Event Sourcing**: ES Order aggregate, replay-based saga, append-only event store, projector rebuild-from-store, rich envelope (`eventId`/`correlationId`/`causationId`/`aggregateVersion`/`schemaVersion`) | State-stored aggregate + Tram transactional outbox; saga state in JDBC `saga_instance`; plain Tram event POJOs via `JSonMapper` (no envelope, no replay/rebuild) | **DD-14** |
| 2 | Order Saga calls OTT `POST /admin/entitlements` as saga step 3 | Dedicated **fulfilment-service** owns post-payment fulfilment; the saga sends one `FulfilOrderCommand`, OTT is one branch inside it | **DD-22, DD-27** |
| 3 | Two-phase billing: Authorize (2) → Provision (3) → **Capture (4, after success)** | **Capture is the pivot, *before* fulfil**; post-pivot failures forward-recover to `CANCELLED_REFUNDED` | **DD-24, DD-26** |
| 4 | Inventory three flavors: Physical / **Slot (repair calendar)** / License; digital infinite | Product types PHYSICAL_GOOD / DIGITAL_SUBSCRIPTION / SOFTWARE_LICENSE (**SLOT removed**); digital finite | **DD-22, DD-23** |
| 5 | Single order flow | **Payment modes** PAY_NOW vs BILL_TO_MOBILE (postpaid check-limit → allocate → next-cycle ledger) added | **DD-23** |

> The original `correlationId`/`causationId` envelope and the "rebuild any read model by replaying the event store" capability are casualties of drift #1 — they no longer exist and are **not** on the backlog (they presuppose ES). Distributed *tracing* (a separate concern) remains pending — see C2.

## 2. Pending backlog — designed in iter 1–3, not built, not superseded

### Section A — Federated auth / OIDC
*OP = **self-hosted Keycloak** (DD-29); the gateway stays **Spring Cloud Gateway** as a resource server, **not** the OP. Implementation in **4 phases**: **A‑1** — stand up Keycloak (realm/clients/users via import) + secure provisioning M2M, incl. audience-restricted tokens (A1 / A4 / A6); **A‑2** — subscriber login via Keycloak, OTT as RP, gated video content (A2 / A3); **A‑3** — gateway edge JWT auth + JWT-derived subject (A5); **A‑4** — production hardening (A7): externalize secrets, TLS, Keycloak prod mode.*

- **A1. OIDC Provider — self-hosted Keycloak** (DD-29): a `keycloak` container + DB and a `vab` realm imported with clients + demo users; standard endpoints at `/realms/vab/protocol/openid-connect/*` + discovery/JWKS. The **gateway is not the OP** — it stays Spring Cloud Gateway as the edge resource server (§A5). *Status: **Phase A‑1 done** — `vab-keycloak` container (Postgres-backed, port 8088) + `vab` realm import with the `vab-provisioning` client (issuer `http://localhost:8088/realms/vab`); the OTT relying-party client + subscriber users land in A‑2. The api-gateway `oauth2-authorization-server` dependency is swapped for `oauth2-resource-server` in A‑3.*
- **A2. OTT as OIDC Relying Party** (against Keycloak): subscriber login (Authorization Code + PKCE), plus OTT's "real" surface — video catalog `GET /v1/videos`, `GET /v1/videos/{id}/stream` gated on the local entitlement. *Status: **Phase A‑2 done** — ott-service is a server-side OIDC **login client** (public `vab-ott` client, Auth Code + PKCE) — **DD-30**. `GET /v1/videos` lists the seeded catalog; `GET /v1/videos/{id}/stream` is session-protected and entitlement-gated — returns `"Playing video: …"` when entitled, **403** otherwise (no real media). A second Spring Security filter chain keeps the §A‑1 provisioning API a Bearer resource server.*
- **A3. OTT token verification**: OTT validates the subscriber's Keycloak token (id_token claims / `userinfo`) before serving gated content. *Status: **done (Phase A‑2)** — the OIDC login establishes the session; `subscriberId` comes from the id-token claim (Keycloak user-attribute mapper) and is matched against ott-service's entitlements.*
- **A4. Client-credentials-secured provisioning** *(was iter-3 design, dropped in DD-27 — now reinstated as backlog)*: secure the fulfilment → OTT call with OAuth2 client-credentials (`scope=ott:provision`) + token caching. *Status: **done (Phase A‑1).** fulfilment fetches a Keycloak client-credentials token (`KeycloakTokenProvider`, cached to ~30s before expiry) and sends it as a Bearer; **ott-service is now an OAuth2 resource server** requiring scope `ott:provision` on `/ott/v1/entitlements/**`. Both provision and revoke are authenticated, and **audience-restricted** — a Keycloak `ott-service` audience mapper puts `aud: ott-service` in the token and ott-service's `AudienceValidator` (alongside the default issuer/expiry checks) rejects tokens minted for any other resource server.*
- **A5. Gateway edge auth (Spring Cloud Gateway as resource server)**: validate Keycloak JWTs (JWKS) + secure the currently-open catalog / orders / entitlements / ops routes; derive `subscriberId` from the token instead of a header/param. *Status: **Phase A‑3 done** — DD-31. The gateway is a reactive **resource server** (JWKS via `issuer-uri`): catalog browse public, orders/entitlements require a valid token, back-office actions (retry / complete / revoke) gated on the `vab-admin` realm role. **Token relay** — the Bearer is forwarded downstream and **order-service is now a resource server too** (defence in depth), reading `subscriberId` from the token `subscriberId` claim (not `sub`, which is a Keycloak UUID; the claim is the A‑2 user-attribute). The `oauth2-authorization-server` dep is dropped for `oauth2-resource-server`.*
- **A6. OIDC scope/claim contract** — configured **in Keycloak** (client scopes + protocol mappers): claims (`sub`, `msisdn_hash`, `name`) and scopes (`openid profile`, `ott:provision`) for the OTT RP and the provisioning client. *Status: **Phase A‑1** — the `ott:provision` client scope (with the `ott-service` audience mapper) is in the realm import; the OTT RP claim mappers (`msisdn_hash`, etc.) come with A‑2. (Moved here from §C4 — federated-auth, not platform.)*
- **A7. Production hardening of the auth stack** (Phase A‑4): externalize secrets — Keycloak admin creds, the `vab-provisioning` client secret, and DB creds — to env / a secret store, removing the hardcoded values from the realm import / `docker-compose.yml` / `application.yml`; **TLS** for Keycloak and the services (`sslRequired`, no bearer tokens over plaintext); run Keycloak in **production mode** (`start`, explicit hostname/proxy config, no dev caching). *Status: none — deferred to Phase A‑4, now **broken out item-by-item into §E** (E1 secrets → E2 TLS → E3 prod-mode). (Audience validation, the other A‑1 hardening gap, is already implemented — see A4.)*

### Section B — Order query surface (iter 3 §2.3 contract)
- **B1. `GET /v1/entitlements` "My Benefits"** — **Implemented (Phase 1–2).** order-service serves `/v1/entitlements?subscriberId=` from a 
   new `entitlements_v1` Mongo projection (the projector activates an entitlement on `OrderCompleted` for DIGITAL_SUBSCRIPTION / SOFTWARE_LICENSE). 
   Uniqueness layer 1 (command-time 409 on an already-active offer) + layer 2 (partial unique index on active `{subscriberId, offerCode}`). 
   Benefit **validity** is threaded end-to-end: catalog `Offer.termMonths` → order snapshots it at placement (fail-open) 
   → fulfilment computes `validFrom`/`validUntil` (now + term) → stored on the OTT entitlement 
   *and* `entitlements_v1` (`validUntil` null = perpetual). 
**Admin revoke (Phase 3):** `POST /v1/orders/{id}/revoke-entitlement` (COMPLETED-only, async via Tram→fulfilment, no refund) 
  → OTT `DELETE` for DIGITAL_SUBSCRIPTION / read-model-only for SOFTWARE_LICENSE → `entitlements_v1` flips to REVOKED, freeing the uniqueness slot. 
  *Remaining: gateway routing + JWT subject (Section A).*
- **B2. `GET /v1/orders/{id}/timeline`** — **Implemented.** Standalone audit-timeline route on `OrderQueryController`, served from `OrderView.timeline` (read-model only; 404 until projected). Full detail with the same timeline embedded stays at `GET /{orderId}`.
- **B3. `order_search_v1`** — **Implemented.** A flattened, timeline-free ops projection built by a **separate** projector (`OrderSearchProjector`) on its own Kafka consumer group (`orderSearchProjector`) — so the same order event stream feeds two independently-rebuildable read shapes (orders_v1 + order_search_v1). Queryable via `GET /v1/ops/orders` (optional `status` / `offerCode` / `from` / `to` filters, newest-first, capped) through `OrderSearchService` (dynamic MongoTemplate query); secondary indexes on `status`/`offerCode`/`placedAt`. *(Gateway routing for `/v1/ops/**` is Section A.)*

### Section C — Cross-cutting / platform (iter 2/3 §7)
- **C1. DLQ per consumer group + replay tool** — *design-only (not built); outline below.*
  - **Gap.** Eventuate messaging-kafka 0.21 has **no DLQ** (verified — only a `BackPressureManager` for flow-control + a `SwimlaneDispatcher` for per-key ordering). A throwing handler isn't acked, so the offset can't advance past it and a **poison message** (schema/deserialization drift, a bug, a permanently-invalid payload) **redelivers forever and head-of-line-blocks** that consumer group. Business failures never reach this path — they return `withFailure(...)` (a *normal* reply) and the saga compensates / forward-recovers (DD-24/26/27); transient technical failures self-heal on redelivery. So DLQ targets only the **non-retryable technical / poison** class — an **operability/resilience** gap, not a correctness one (hence low priority for the demo).
  - **Broker: Kafka-native, *not* RabbitMQ.** The whole backbone is Kafka + the Tram **outbox → CDC → Kafka** path. Rabbit's nicer native dead-letter-exchanges aren't worth standing up a **second broker** bolted onto only the consumer edge (split topology, dual ops, no outbox guarantee on the Rabbit side). Trade-off accepted: Kafka has **no native DLX**, so the DLQ is a **pattern we build** (retry → produce to a `.DLT` topic). Spring-Kafka / Spring-Cloud-Stream DLQ helpers don't apply — Eventuate runs its **own** consumer, not `@KafkaListener`.
  - **Mechanism — a `MessageHandlerDecorator`** (the same SPI as `CorrelationMessageInterceptor` and Eventuate's tracing decorator — already proven in this codebase): wrap `chain.invokeNext`, catch, apply **bounded retry + backoff** (ideally classifying retryable vs poison), and on exhaustion **produce the message to a per-group DLT and let the original ack** so the swimlane unblocks.
  - **Components.** (1) a `DeadLetterMessageHandlerDecorator`; (2) **per-group DLT** topics `<channel>.<subscriberId>.DLT` — so a fanned-out event only DLQs the *group* that fails, not the others; (3) a **retry-count store** — Kafka re-reads the immutable record on redelivery, so the attempt count lives in a small `message_retry(message_id, subscriber_id, attempts)` table (or an extension of `received_messages`), **not** on the message; (4) DLT records that **preserve headers** (correlationId, `traceparent`, saga id) + add failure metadata (exception, source topic/partition/offset, attempts, timestamp); (5) a **replay tool** (admin endpoint / CLI) that re-publishes a DLT record to its origin channel with headers intact; (6) a `dlq.messages` **metric + Grafana alert** wired into the C2 Loki/Tempo stack — a DLT nobody watches is a silent black hole.
  - **Key traps.** *Ordering vs progress:* skipping a poison message to unblock the partition **breaks per-key ordering** — fine for **projection/event** consumers (a missed read-model update, replay later), but **dangerous on the saga command/reply path** (skip a command and the saga **hangs** waiting for a reply). So true skip-DLQ belongs on the **projections first**; the **saga path** is better treated as *capture + alert + manual re-drive*, leaning on the existing **DD-27 park/re-drive** + saga timeouts rather than blind skip. *Deserialization* failures happen **before** the handler, so a handler-wrapping decorator can't catch schema-drift poison — that needs the consumer/deserializer layer. **Idempotency is already satisfied** (Tram `received_messages` dedup + business-level idempotency), so redelivery *and* replay are safe — the key enabler.
  - **Suggested scope + effort.** Phase 1 — projection/event consumers (safe to skip → DLT) with bounded retry + per-group DLT + replay endpoint + alert. Phase 2 — saga path as alert + manual re-drive on the DD-27 machinery. Rough effort: decorator + DLT + retry-state + replay + per-service wiring + tests ≈ **a few focused days**; retry-state persistence and the projection-vs-saga split are the fiddly parts. *Status: none (documented, deferred).*
- **C2. Distributed tracing / correlation**: W3C `traceparent` propagated over HTTP + Kafka headers, `correlationId`/`causationId`, OTel exporter → Loki/Grafana panels. *Status: **B‑1 done** — chose **scope B** (correlation + structured logs + Loki/Grafana; no OTel spans). Every service emits **structured JSON** to console and **pushes to Loki** (loki4j direct push — adapted from the `eventuate-saga` repo's OTel-Collector approach, which assumes containerised stdout; ours run on the host), keeping the human-readable `vabags-<svc>.log` files. Compose adds **Loki** (`:3100`) + **Grafana** (`:3000`, anon-admin) with a provisioned datasource + "VA-BAGS — Logs" dashboard.  **B‑2 done** — a new **`shared-observability`** module (auto-configured): a servlet **`CorrelationFilter`** seeds `correlationId` into MDC at the HTTP edge, and a Tram **`CorrelationMessageInterceptor`** carries it across Kafka (MDC → message header on `preSend`, header → MDC on `preHandle`, cleared on `postHandle`; `causationId` = the handled message's id). The header rides the transactional outbox so it survives the CDC → Kafka hop. Wired into the 5 saga services (order/inventory/billing/fulfilment/notification); the **gateway** originates/forwards `X-Correlation-Id` (reactive `GlobalFilter`). `correlationId` is in the JSON console logs *and* the Loki push, and the Grafana dashboard gained a **`$correlationId`** filter — set it to follow one order across every service. *Verified the Tram `MessageInterceptor`/`Message` API against 0.37.0 before wiring (`preSend`/`preHandle`/`postHandle`, `setHeader`/`getHeader`).* The **internal HTTP hops are also linked** — `CatalogClient` (order→catalog) and `OttClient` (fulfilment→ott) forward `X-Correlation-Id` from MDC, and catalog/ott carry the `CorrelationFilter`, so their logs join the same trail. **Scope C done** — distributed tracing via Micrometer → OTel bridge + OTLP → **Tempo**. The hard part (Kafka/saga `traceparent` propagation across the outbox) uses Eventuate's **ready-made `eventuate-tram-spring-micrometer-tracing-starter`** (its own producer/consumer `MessageInterceptor`s — *no hand-rolled span code*). Sampling `1.0` at the roots (gateway/order); the decision rides `traceparent` downstream. `traceId`/`spanId` are auto-added to MDC → JSON logs + Loki push, and Grafana wires **Loki↔Tempo** both ways (a log line's `traceId` links to its trace; a trace links back to its logs). `correlationId` is **kept** alongside as the tracing-independent, vendor-neutral, client-facing id (see the keep-vs-remove note — they serve different layers). *The CDC hop shows as a time-gap in the waterfall — inherent to tracing over the outbox, not a bug.*
  - *Fragmentation post-mortem:* traces first appeared **fragmented** (one trace per saga hop). Root cause was **not** a library gap — the participant command handlers ran with **no active span** (`traceId` blank in their logs) because that service's build was **stale**: the `eventuate-tram-spring-micrometer-tracing` starter wasn't on its runtime classpath, so its three auto-configs never registered (confirmed via `/actuator/conditions` + `beans` — `observationHelper` / `observationMessageConsumerDecorator` / `observationMessageProducerInterceptor` all absent, only the 3 built-in `MessageHandlerDecorator`s in the chain). A **clean rebuild** put the starter on the classpath and the trace became one connected waterfall end-to-end. Verified against the library source: the consumer decorator (`ObservationHelper.startConsumerObservation`) correctly extracts `traceparent` + opens the parent-span scope — it just has to be *loaded*.
  - *Loki line body is structured JSON:* the loki4j `<message>` now uses logstash `LogstashLayout`, so `message` / `traceId` / `spanId` / `correlationId` / `logger_name` / `thread_name` are first-class **`| json`** fields (loki4j 1.5.2 has no structured-metadata support, so a JSON body is the way to get typed, filterable fields). Loki's `discover_service_name` is disabled to avoid a duplicate `service_name` label; the Grafana derived-field regex matches the JSON `"traceId"`.
  - *Local-run gating (perf, 16 GB box):* the whole trace/log stack is **opt-in** via two Spring profiles. **`obs`** → Loki push (`<springProfile name="obs">` on the `LOKI` appender) **and** OTLP export to Tempo (`management.otlp.tracing.export.enabled=true`), paired with `COMPOSE_PROFILES=obs` for the `loki`/`grafana`/`tempo` compose services. **`jsonlogs`** → JSON console instead of the plain Spring pattern. **Default (neither):** plain console, no Loki, **no export** — but `traceId` is still generated & propagated (roots sample `0.0`; a non-recording span still has a valid id) so it lands in the per-service `vabags-<svc>.log` files and could be shown anywhere. Export is **hard-gated** by `management.otlp.tracing.export.enabled=false` (not sampling) — because the parent-based sampler honours a sampled `traceparent` arriving on an old Kafka message, so only killing the exporter bean reliably stops stray Tempo connection attempts. Kafka client chatter is dropped to `WARN`; the console trace-correlation is blanked (traceId stays in files/Loki).
- **C3. Schema-registry-backed event versioning** *(was iter-2/3 design — now backlog)*: register `<aggregate>.<Event>.vN` JSON-Schemas in Apicurio with additive-only evolution. *Status: the Apicurio container runs but is **unused** — events are plain Java classes serialized via `JSonMapper`.*
- **C4. Richer eligibility-rule shape** (catalog): eligibility beyond the current plan-tier / region / device-age / KYC dimensions. *Status: deferred since iter 3 §10.* *(The OIDC scope/claim contract previously bundled here moved to §A6 — it's federated-auth, not platform.)*

### Section D — Smaller deferrals (from the DD log)
- **D1. Catalog domain events + price-snapshot consumer** — **deliberately not built (not an oversight).** The *correctness* goal ("historical orders carry a snapshot, not a live FK") is **already satisfied**: an order copies `amount`/`currency` at placement and is immune to later catalog edits. The remaining pieces are unneeded here — snapshotting the display *name* is **cosmetic** (no UI consumes it; e2e asserts status, not names), and the catalog **event mechanism** (`OfferPublished` / `OfferWithdrawn` / `PriceChanged` → cross-service mirror) is only justified by a real consumer, which doesn't exist (DD-17 deemed its Mongo-publication infra not worth the cost). *Revisit only if a UI needs frozen names or a consumer needs a live catalog mirror. Ref: DD-17 / Design/02.*
- **D2. `abandon → CANCELLED_REFUNDED` escape hatch** from `FULFILMENT_FAILED`. *Deferred: DD-27.*

### Section E — Production hardening (A7 / Phase A‑4, prod-pending)
*The auth stack works but carries local-demo shortcuts. These are the prod-readiness items broken out of §A7 and sequenced **secrets → TLS → Keycloak prod-mode**, because each depends on the previous (prod-mode needs TLS).*

- **E1. Externalize secrets** — move real creds out of source control:
  - `KC_BOOTSTRAP_ADMIN_PASSWORD` and `KC_DB_USERNAME/PASSWORD` in `docker-compose.yml`;
  - the DB `password` in every service `application.yml`;
  - the `vab-provisioning` **client secret** (declared in `vab-realm.json`, consumed in fulfilment `application.yml`).

  *Approach (agreed):* store = a **gitignored `.env`** consumed by `docker-compose`, with a committed **`.env.example`** of placeholders; services read `${DB_PASSWORD}` / `${VAB_PROVISIONING_SECRET}` (Spring already supports this). The realm-import client secret **stays in `vab-realm.json` as a known dev secret** (Keycloak import doesn't substitute env vars) but is env-referenced everywhere it's *consumed*; a post-import `kcadm` rotation is the real-prod path. Demo user passwords (`alice`, `vabadmin`) are **kept as fixtures**, not externalized. A real deployment swaps `.env` for a **vault / cloud secret manager**.

  *Change is behaviour-neutral.* **Sanity after implementation (no new tests needed):** re-run **`OttAuthE2E.authenticated_provision_succeeds`** (the `vab-provisioning` client-credentials grant — 401 if the secret env-wiring is wrong) and **`OrderHappyPathE2E.digital_subscription_pay_now_completes_with_external_ref`** (end-to-end proof that the externalized DB + Keycloak-admin + provisioning secrets all still resolve). *Status: pending.*

- **E2. TLS — edge-only, implemented.** Decisions (agreed): **edge TLS only** (gateway + Keycloak serve HTTPS; internal service↔service stays HTTP on the trusted local net), certs via **mkcert** (locally-trusted CA → no per-service truststore edits), **Keycloak built-in** HTTPS (no reverse proxy), realm `sslRequired: **external**`. *What changed:* Keycloak publishes HTTPS on `:8088` (container `:8443`, mkcert cert mounted); the gateway serves HTTPS on `:8089` (`server.ssl`, PKCS12 keystore); all `issuer-uri`/`token-uri` → `https://localhost:8088`; realm `sslRequired=external`; e2e points at the HTTPS URLs + `useRelaxedHTTPSValidation()`. Cert generation is a **one-time manual step** ([`deploy/tls/README.md`](../deploy/tls/README.md)) — Keycloak/gateway won't boot without the cert files. Residual: **ott-service stays HTTP** (browser session over HTTP; credentials themselves go to Keycloak over HTTPS) — acceptable for the demo. **Sanity:** the whole `-Pe2e` suite (now over HTTPS to gateway+Keycloak) plus the OTT browser-login flow (`OttVideoE2E`). *Status: **implemented**, but now **opt-in behind a flag** (see below).*
  - *Update (§4, local-run default = HTTP dev):* to cut startup friction on the dev box, **Keycloak now defaults to plain HTTP** `start-dev` (`:8088`→ container `:8080`, no certs), and the app/e2e `issuer-uri`/`token-uri`/`vab.keycloak.url` **defaults flipped back to `http://…`**. The edge-TLS setup is layered back on **on demand** via the override file `docker-compose.keycloak-tls.yml` — `KC_HTTP_PORT=8443 docker compose -f docker-compose.yml -f docker-compose.keycloak-tls.yml up -d keycloak` (adds the cert env + mounts) plus `KEYCLOAK_ISSUER=https://…` / `KEYCLOAK_TOKEN_URI=https://…` on the apps and `-Dvab.keycloak.url=https://…` on e2e. The **gateway keeps its own HTTPS on `:8089` regardless** (that's the edge, unchanged). Realm `sslRequired=external` works for both — local/private-network requests are exempt, so plain HTTP is allowed on the dev box. mkcert cert generation stays the one-time manual step only for the TLS path.
- **E3. Keycloak production mode** — `start` (not `start-dev`), explicit `KC_HOSTNAME`/proxy config, `--optimized` build, no dev caching. Depends on E2. *Status: pending.*

## 3. Built & matches the iter 1–3 intent
Idempotency — client-supplied key + `idempotency_keys` table (DD-07); single-deployable CQRS order-service (DD-06); saga orchestration + LIFO compensation + forward-recovery (now via Tram, not ES); Mongo `orders_v1` projection with embedded timeline; gateway routing for catalog/orders; the three product types, payment modes, and OTT provisioning + park / admin re-drive (DD-27).
